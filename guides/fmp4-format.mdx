Each render chunk is delivered as a **fragmented MP4** (fMP4) segment. This format enables progressive download, Media Source Extensions (MSE) playback, and byte-concatenation into a complete video.

## Segment Structure

Chunks follow the W3C ISO BMFF Byte Stream Format for streaming compatibility:

```
Chunk 0 (init + first segment):
  [ftyp][moov][moof₁][mdat₁]

Chunk 1+ (media segments only):
  [moof₂][mdat₂]
  [moof₃][mdat₃]
  ...
```

- **Chunk 0** contains the initialization segment (`ftyp` + `moov`) plus the first media segment
- **Chunk 1+** contain only media segments (`moof` + `mdat`) — no redundant init
- **No `mfra`** (Movie Fragment Random Access) — allows seamless streaming append

## Codec

| Property | Value |
|----------|-------|
| Video codec | H.264 (AVC) Main Profile |
| Encoder | Hardware-accelerated (h264_nvenc) |
| Frame rate | 25 fps (constant) |
| Keyframe | IDR at the start of each chunk |
| B-frames | None (`bf=0`) |
| PTS/DTS | Identical (no composition time offsets) |
| Timescale | 12800 ticks/second (512 ticks per frame at 25fps) |

<Info>
  The codec string for MSE `addSourceBuffer()` is typically:
  `'video/mp4; codecs="avc1.4d0028"'` (Main profile, level 4.0)
</Info>

## Box Layout

### Chunk 0 (Init + First Segment)

```
ftyp                      File Type Box
├─ major_brand: isom
└─ compatible: isom, iso2, avc1, mp41

moov                      Movie Box (empty — no sample tables)
├─ mvhd                   Movie header (timescale, duration=0)
├─ trak                   Track Box
│  ├─ tkhd                Track header (width, height, track_id=1)
│  └─ mdia                Media Box
│     ├─ mdhd             Media header (timescale=12800)
│     ├─ hdlr             Handler (vide)
│     └─ minf → stbl      Sample table (empty: stts, stsc, stsz, stco)
└─ mvex                   Movie Extends Box
   └─ trex                Track extends defaults

moof                      Movie Fragment (sequence_number=1)
├─ mfhd                   Fragment header
└─ traf                   Track Fragment
   ├─ tfhd                Track fragment header (default_base_is_moof=true)
   ├─ tfdt                base_media_decode_time=0
   └─ trun                Track run (25 samples, no composition offsets)
mdat                      Media Data (H.264 NAL units for 25 frames)
```

### Chunk 1+ (Media Segments)

```
moof                      Movie Fragment (sequence_number=N+1)
├─ mfhd                   Fragment header
└─ traf                   Track Fragment
   ├─ tfhd                default_base_is_moof=true
   ├─ tfdt                base_media_decode_time=N*12800 (N seconds)
   └─ trun                Track run (25 samples)
mdat                      Media Data
```

## Key Fields

| Field | Location | Description |
|-------|----------|-------------|
| `sequence_number` | moof → mfhd | Fragment index (1-based, monotonically increasing) |
| `base_media_decode_time` | traf → tfdt | Decode timestamp in timescale units. Chunk N starts at `N * 25 * 512 = N * 12800` |
| `default_base_is_moof` | traf → tfhd flags | Data offsets relative to `moof` position (required for byte-concatenation) |
| `sample_count` | traf → trun | Number of frames (typically 25 per chunk, last chunk may have fewer) |
| `sample_duration` | trun entries | 512 ticks per frame (= 12800 / 25fps) |
| `sample_composition_time_offset` | trun entries | **Always 0** — PTS equals DTS for every frame |

## Byte-Concatenation

Chunks are designed for direct byte-concatenation into a playable video:

```
cat chunk_0.mp4 chunk_1.mp4 chunk_2.mp4 ... > complete.mp4
```

This works because:
1. Only chunk 0 has `ftyp` + `moov` (initialization)
2. `default_base_is_moof` makes data offsets relative (not absolute file positions)
3. `sequence_number` increments across chunks (1, 2, 3...)
4. `base_media_decode_time` is continuous (0, 12800, 25600...)
5. No composition time offsets — PTS == DTS

The concatenated result is a valid fMP4 that can be remuxed to standard MP4:

```bash
ffmpeg -y -i concatenated.mp4 -c copy -movflags +faststart output.mp4
```

## MSE Playback

For browser playback via [Media Source Extensions](/guides/live-render#progressive-playback-with-fmp4--mse):

```javascript
const mimeType = 'video/mp4; codecs="avc1.4d0028"';
const sourceBuffer = mediaSource.addSourceBuffer(mimeType);

// Append each chunk as it arrives from the poll endpoint
for (const chunk of newChunks) {
  const data = await fetch(chunk.cdn_url).then(r => r.arrayBuffer());
  sourceBuffer.appendBuffer(data);
  await new Promise(r => sourceBuffer.addEventListener('updateend', r, { once: true }));
}
```

<Warning>
  Append chunks **in order** (chunk 0 first, then 1, 2, ...). Chunk 0 contains the initialization segment that MSE needs before it can decode any media segments.
</Warning>

## Final Merged Video

The download endpoint (`GET /api/render/{job_id}/download`) returns a **standard MP4** (not fMP4) with `+faststart`. This is the recommended format for storage and distribution.

## Verification

```bash
# Show codec and format
ffprobe -v quiet -show_streams -show_format chunk_0.mp4

# Check PTS == DTS (no composition offsets)
ffprobe -show_packets -select_streams v chunk_0.mp4 | head -20

# Inspect box layout (requires Bento4)
mp4dump chunk_0.mp4
```
